# RPEngine Character Transport Standard

## 1. Purpose

RPEngine uses **Character Card V2** as the canonical format for transporting a character from a game integration to the character roleplay engine.

This standard deliberately separates two concepts:

1. **Character document** — a complete Character Card V2 JSON object.
2. **Transfer descriptor** — the snapshot, patch, or reference wrapper used to deliver that document efficiently.

Games should not define a separate RPEngine-specific character schema. Game state that describes the character should be mapped into Character Card V2 fields or namespaced extension fields.

The authoritative Character Card V2 definition is the vendored [`spec_v2.md`](../spec_v2.md), sourced from the [Character Card V2 specification](https://github.com/malfoyslastname/character-card-spec-v2/blob/main/spec_v2.md).

## 2. Canonical character document

Every reconstructed card MUST be a JSON object with:

- `spec` equal to `"chara_card_v2"`.
- `spec_version` equal to `"2.0"`.
- `data` containing all required Character Card V2 fields with their standard types.
- A non-empty `data.name`.

Required fields that have no game-specific value MUST still be present. Use an empty string, empty array, or empty object as appropriate. Producers MUST NOT omit required fields.

```json
{
  "spec": "chara_card_v2",
  "spec_version": "2.0",
  "data": {
    "name": "Ari",
    "description": "A cautious engineer responsible for the station reactor.",
    "personality": "Observant, practical, loyal, and quietly funny.",
    "scenario": "A damaged research station is drifting without outside support.",
    "first_mes": "{{char}} looks up from the diagnostic console. \"What happened?\"",
    "mes_example": "<START>\n{{user}}: Is the reactor stable?\n{{char}}: Stable enough for now. I would still keep the emergency shutters closed.",
    "creator_notes": "Generated by the example game integration.",
    "system_prompt": "Write the next response from {{char}} in character.",
    "post_history_instructions": "Remain consistent with the supplied character and current event.",
    "alternate_greetings": [],
    "tags": ["engineer", "science fiction"],
    "creator": "Example Game",
    "character_version": "1.0.0",
    "extensions": {}
  }
}
```

RPEngine preserves unknown fields and extension data. Integration-specific extensions SHOULD use a stable namespaced key to avoid collisions:

```json
{
  "extensions": {
    "com.example.game/character_state": {
      "faction": "Station Crew",
      "health": "injured"
    }
  }
}
```

Optional standard fields such as `data.character_book` remain part of the canonical document. Transport support does not imply that every optional field affects the current inference runtime. RPEngine presently preserves `character_book` but does not perform lorebook scanning or activation.

## 3. Macros

Character Card V2 prompt text may use:

- `{{char}}` — replaced with `data.name`.
- `{{user}}` — replaced with `player.displayName`, or `Player` when no display name is supplied.
- `{{original}}` — follows Character Card V2 replacement semantics for supported prompt fields.

Games SHOULD send the template form of the card. RPEngine owns macro rendering during prompt assembly.

## 4. Canonical hash

Each transfer descriptor identifies its resulting character document with `targetHash`.

The hash MUST be calculated as:

1. Serialize the complete Character Card V2 document using RFC 8785 canonical JSON.
2. Encode the canonical JSON as UTF-8.
3. Calculate SHA-256 over those bytes.
4. Encode the digest as 64 lowercase hexadecimal characters.

Hash the complete outer document, including `spec`, `spec_version`, `data`, standard optional fields, unknown fields, and extensions. Do not hash only the `data` object.

The producer and RPEngine MUST calculate the same hash before a snapshot or reconstructed patch is accepted.

## 5. Transfer descriptors

All transfer descriptors use:

```json
{
  "format": "chara_card_v2",
  "mode": "snapshot | patch | reference"
}
```

### 5.1 Snapshot

A snapshot carries the complete canonical document. Use it for the first request involving a character, after reconnecting, or after a resynchronization error.

```json
{
  "format": "chara_card_v2",
  "mode": "snapshot",
  "snapshot": {
    "spec": "chara_card_v2",
    "spec_version": "2.0",
    "data": {
      "name": "Ari",
      "description": "A cautious engineer.",
      "personality": "Practical and loyal.",
      "scenario": "A damaged research station.",
      "first_mes": "",
      "mes_example": "",
      "creator_notes": "",
      "system_prompt": "",
      "post_history_instructions": "",
      "alternate_greetings": [],
      "tags": [],
      "creator": "Example Game",
      "character_version": "1.0.0",
      "extensions": {}
    }
  },
  "targetHash": "0123456789abcdef0123456789abcdef0123456789abcdef0123456789abcdef"
}
```

RPEngine validates the card and verifies `targetHash` before caching it.

### 5.2 Patch

A patch carries RFC 6902 JSON Patch operations relative to a previously accepted card.

```json
{
  "format": "chara_card_v2",
  "mode": "patch",
  "baseHash": "0123456789abcdef0123456789abcdef0123456789abcdef0123456789abcdef",
  "patch": [
    {
      "op": "replace",
      "path": "/data/scenario",
      "value": "The station reactor has entered emergency shutdown."
    },
    {
      "op": "add",
      "path": "/data/tags/-",
      "value": "emergency"
    }
  ],
  "targetHash": "abcdef0123456789abcdef0123456789abcdef0123456789abcdef0123456789"
}
```

The producer MUST:

1. Build the new complete card locally.
2. Create the patch against the exact card identified by `baseHash`.
3. Calculate `targetHash` from the complete patched document.

RPEngine applies the patch to a clone of the cached base, validates the reconstructed V2 card, and verifies `targetHash`. A patch MUST NOT remove a required field or change it to an invalid type.

Supported RFC 6902 operations are `add`, `remove`, `replace`, `move`, `copy`, and `test`.

### 5.3 Reference

A reference indicates that the accepted card has not changed:

```json
{
  "format": "chara_card_v2",
  "mode": "reference",
  "targetHash": "abcdef0123456789abcdef0123456789abcdef0123456789abcdef0123456789"
}
```

Use a reference only after RPEngine has acknowledged the same hash during the current connection session.

## 6. Cache scope and lifecycle

RPEngine keys reconstructed cards by:

```text
integrationId + characterId
```

The cache is scoped to the authenticated connection session. It is not a persistent character database.

- `integrationId` identifies the game or mod integration.
- `characterId` is the integration's stable identifier for that character.
- Reconnecting starts a new synchronization lifecycle.
- The first transfer in a new session SHOULD be a snapshot.
- A successful snapshot or patch replaces the cached clone for that key.

## 7. Proactive synchronization

`character.sync` updates the cached card without requesting inference:

```json
{
  "protocol": "gemtavern.rp_engine",
  "protocolVersion": 3,
  "type": "character.sync",
  "messageId": "message-001",
  "sessionId": "active-session-id",
  "timestamp": "2026-07-15T00:00:00.000Z",
  "integrationId": "com.example.game",
  "characterId": "character-42",
  "card": {
    "format": "chara_card_v2",
    "mode": "snapshot",
    "snapshot": {
      "spec": "chara_card_v2",
      "spec_version": "2.0",
      "data": {
        "name": "Ari",
        "description": "A cautious engineer.",
        "personality": "Practical and loyal.",
        "scenario": "A damaged research station.",
        "first_mes": "",
        "mes_example": "",
        "creator_notes": "",
        "system_prompt": "",
        "post_history_instructions": "",
        "alternate_greetings": [],
        "tags": [],
        "creator": "Example Game",
        "character_version": "1.0.0",
        "extensions": {}
      }
    },
    "targetHash": "0123456789abcdef0123456789abcdef0123456789abcdef0123456789abcdef"
  }
}
```

On success, RPEngine returns `character.synced` with the accepted target hash.

## 8. Character transport in a reply request

Every `reply.request` includes a transfer descriptor. This avoids requiring a separate synchronization round trip before inference:

```json
{
  "protocol": "gemtavern.rp_engine",
  "protocolVersion": 3,
  "type": "reply.request",
  "messageId": "message-002",
  "sessionId": "active-session-id",
  "timestamp": "2026-07-15T00:00:01.000Z",
  "requestId": "request-001",
  "eventId": "event-901",
  "integrationId": "com.example.game",
  "characterId": "character-42",
  "event": {
    "text": "The reactor alarm begins sounding. Player asks Ari what is wrong."
  },
  "output": {
    "modalities": ["text"],
    "language": "en"
  },
  "player": {
    "displayName": "Player"
  },
  "card": {
    "format": "chara_card_v2",
    "mode": "reference",
    "targetHash": "0123456789abcdef0123456789abcdef0123456789abcdef0123456789abcdef"
  }
}
```

On acceptance, `reply.accepted` identifies the resolved card hash used for that request.

## 9. Errors and recovery

RPEngine returns `request.error` with:

- `invalid_character_card` when a snapshot or reconstructed patch is not a valid Character Card V2 document.
- `card_resync_required` when the base/reference is unavailable, a base hash does not match, patch application fails, or the reconstructed hash does not match `targetHash`.

When `card_resync_required` is received, the game SHOULD retry the same logical request with:

- The same `requestId` and `eventId`.
- A new message envelope and unique `messageId`.
- A full snapshot containing the current complete card.

Do not retry a malformed card unchanged after `invalid_character_card`; correct the document first.

## 10. Producer requirements

A conforming game integration MUST:

1. Produce a complete Character Card V2 document.
2. Preserve correct JSON types for all standard fields.
3. Calculate the canonical SHA-256 hash over the complete card.
4. Send a snapshot when no accepted session base exists.
5. Generate patches only from the exact acknowledged base card.
6. Resend a snapshot when RPEngine requests resynchronization.
7. Treat card hashes as content identifiers, not character identifiers.
8. Keep `integrationId` and `characterId` stable for the life of the game character.

An integration MAY always use snapshots. Patch and reference modes are bandwidth optimizations, not requirements for basic compatibility.
